## Summary

All three flags and the worst qa_log rows share one signature: the model correctly names
exactly what's missing (figure layouts, sub-sections), and the agentic loop fails to act
on that information. It's not a data problem — figures 630, 631, 632, 633, and 642 all
exist in data/tables.json and the chunk index. The failures are four interacting logic
gaps in orchestrator.py's agentic loop, plus one already-half-fixed retrieval gap.

## Root Causes (ranked by impact)

### BUG 1 — First-wins dedup blocks pin promotion [HIGHEST PRIORITY]
**File:** orchestrator.py:1361
**Symptoms:** False stalls; figures fetched but never pinned; loop declares "context
unchanged" while the answer remains incomplete.

The agentic merge at orchestrator.py:1361 dedups `expanded_pool + extra_chunks` keeping
the *first* copy. A chunk first seen via hybrid search keeps method='rrf' forever. When
the targeted fetch later re-fetches the same chunk id with a pinnable method
(agentic_fetch_*), the new copy is silently dropped, so _pin_structured_hits
(orchestrator.py:272) never pins it, and the cross-encoder keeps cutting it (byte-layout
tables score poorly against prose queries).

Hard evidence in flag 19: fig630__ was retrieved by follow-up hybrid search in iter1
(BM25 score 25.8, vector 0.77) and entered the pool as 'rrf'; fig632__ was
targeted-fetched in iter2 with a pinnable method — yet the loop declared "reranked
context unchanged" and stalled.

Fix shape: on dedup collision, promote the kept chunk's method if the incoming copy's
method is pinnable.

---

### BUG 2 — verdict.missing field never fed back into retrieval
**File:** orchestrator.py:667, 1223
**Symptoms:** Model names the exact missing figures/sections; cheap gap-analyzer
re-derives a worse list from a truncated answer; named resources never fetched.

_agentic_gap_analysis (orchestrator.py:667, called at 1223) receives only the query,
section titles, hallucinated cites, and answer[:1500] — the verdict is parsed and then
used only as a boolean stop signal at line 1208. The strong model enumerates the shopping
list; the cheap gap-analyzer re-derives a worse one from a truncated answer.

Flag 19 evidence: iter1 regenerate verdict said verbatim:
  missing: "Figure 630/631/632/642 layouts, DHK derivation (8.1.6.3.1.1),
            Random Nonce Request (8.1.6.3.2)"
The iter2 gap analysis requested none of fig630/631/642 and not 8.1.6.3.2.

Related: the 1500-char truncation at orchestrator.py:698 hides "missing information"
caveats these answers put at the *end*.

Fix shape: feed verdict.missing into the next iteration's requested_resources.
_parse_requested_resources could almost consume it directly as it's already structured
text naming figures/sections.

---

### BUG 3 — Convergence check ignores verdict.missing; answered=true with non-empty missing wrongly converges
**File:** orchestrator.py:1208
**Symptoms:** "It stops while admitting it lacks context" (flags 20, 21).

Line 1208: `if verdict.get("answered") and not unfetched_cites: converged`
Flag 20's verdict was answered=true, missing="Byte-level layout of Figures 632/633" —
a self-contradiction the loop accepts. It only kept iterating because of an unrelated
hallucinated cite, then stalled (bug 1) and ran the context_is_final wrap-up, producing
exactly the flagged behavior: an answer that admits incompleteness while fetchable
figures sat unfetched.

Fix shape: a non-empty missing field that names concrete figure/section ids should block
convergence and seed the next fetch (which is bug 2's fix).

---

### BUG 4 — Hallucinated-cite fetching has no memory and no "unfetchable" escape
**File:** orchestrator.py (agentic loop, unfetched_cites logic)
**Symptoms:** Same cite chased every iteration; loop can only exit via stall.

Flag 20 chased 8.1.6.3.1.1 in both iterations; fetch_section_chunks returned the parent
8.1.6.3.1__c0 both times, the regenerated answer still cited 8.1.6.3.1.1, so
unfetched_cites stayed non-empty, convergence stayed blocked, and the only exit was the
stall. Every iteration of every flag also re-fetched fig285 even though it was already
in context.

Fix shape: per-request "already tried" set; a cite that survives a fetch attempt should
be treated as unresolvable rather than blocking convergence forever.

## Secondary Observations

- Figure-ref expansion (_expand_referenced_figures, committed yesterday in cbcc018)
  directly targets flag 21's complaint (prose says "refer to Figure 630" and the layout
  never arrives) — all three flags predate it. Two caveats: its fetched chunks get id
  agentic_fetch:<spec>:figN while the indexed copy of the same table is figN__, so the
  dedup pool can hold both (duplicate context, and bug 1's collision in reverse); and
  it's not in the _PINNED set, which is fine for append-mode but means an expansion
  chunk in the pool can't survive the next rerank.

- [DONE] Verdict parse returned null once (flag 19 iter0, Opus). The @@VERDICT@@ block trails
  the answer, so a max_tokens truncation eats it; worth checking stop_reason when
  verdict is null and the answer is long. (Fixed: generate() now synthesizes an
  answered=false verdict when emit_verdict + stop_reason=max_tokens + no verdict line,
  so the agentic loop treats truncated answers as incomplete.)

- qa_log id 86 is user error, not pipeline: an NVMe PKAS question asked with spec=pcie.
  A cross-spec "did you mean" hint would catch it, but it's not a pipeline failure.

- Non-agentic rows (ids 4, 5, 38) that name a missing section are working as designed —
  the gap_hint fires and waits for the user to click refine.
