## Systematic issues

1. **Summary generation is ~65% complete and state file lies about it.** 364 cards have empty `summary` (35% of the corpus), yet `cards_state.processed` contains all 1041 section IDs. Level breakdown of the gap: L1=6/12, L2=23/69, L3=150/308, L4=142/376, L5=23/225, L6=10/10 (all), L7=10/41. The empty-summary set and empty-keywords set are identical, so keywords appear to be a by-product of the summary LLM call. Downstream consequences: (a) §1.7 chunking prepends the card summary before embedding — 35% of chunks will have no summary to prepend; (b) L6 is 100% empty; (c) top-level L1 cards `1`, `2`, `9`, … missing summaries is especially damaging since they anchor topic retrieval. Fix: re-run generation, and have the state file store `{status, error, model, timestamp}` per section rather than a flat processed-list.

2. **Duplicate `5.3.1` drops content silently.** The TOC duplicate-key bug (documented in `02_toc_accuracy.md`) propagates downstream with asymmetric damage: `toc.json` and `prose.json` keep both entries (second `5.3.1` has `paragraphs: []` and no page range), but `cards.json` collapses to a single card for "Command Completion" and loses any handle on "Create I/O Completion Queue command". 10 relationships touch `section:5.3.1` — they now target whichever entry happened to be indexed first. Fix must happen at the TOC layer (§1.1) and then rebuild downstream; downstream cannot untangle this alone.

3. **`cards.tables` is severely undercounted.** Only 203 of the 717 table→section links captured by `relationships.json` `contained_in` edges are mirrored in `cards.tables`. The two views disagree for 71.7% of tables. Either (a) the card builder's table attribution logic uses different rules (e.g., only tables whose caption explicitly names the section), or (b) the two pipelines ran against different intermediate states. Downstream consequence: §1.7 chunking and graph-build will wire tables to the wrong cards, and retrieval over "all tables in section X" will silently miss most of them. Fix: single-source the section→figure map from `relationships.json`.

4. **Relationship orphan rate 3.9% (186/4820).** Two distinct failure modes:
   - **Figure-target orphans (131).** Point to figures beyond the parsed set — e.g., `figure:706`, `figure:670`, `figure:199`. Cross-checked: `figure:706` is referenced by 5+ sources with strong textual evidence ("Refer to Figure 706") but does not exist in `tables.json`. Table parser missed these figures, not the relationship extractor. This is the "never hallucinate; expose gaps honestly" bar from §1.8 being violated silently.
   - **Section-target orphans (55).** Point to deep sections like `5.2.26.1.24.3`, `3.3.2.1.3.1`, `8.1.6.3.1.1`. These are either body-text references to sub-sub-sections that the TOC parser never captured, or hallucinated depth. Either way, any graph traversal through these edges dead-ends.

5. **Phantom TOC children confirmed downstream.** Exact parent-title-duplicate cards: `B.5.1.1` ("B.5.1. Shadow Doorbell Buffer Overview"), `B.6.1.2` ("B.6.1. Write after Write"), `B.6.2.1` ("B.6.2. Non-Idempotent Command"), `B.6.4.2` ("B.6.4. Retried Command Affects Another Host"). Plus `B.6.1.1` carries `B.5.2. Example Algorithm...` — a cross-contamination, not just a duplicate. All 6 deep annex-B cards (B.5.1.1 through B.6.4.2) are bogus inventions that will generate duplicate or misleading retrieval hits.

6. **Hierarchy shift in §3.2 / §3.3 is baked into cards.** Card `3.2.1.1` carries the register-offset title "Offset E18h: PMRMSCU …" rather than the authoritative "Namespace Overview" from the PDF outline. Because parent_section / child_sections chains are internally consistent in cards.json (Issue #0 in the good column), the wrong titles and misattributed prose/tables propagate cleanly down the tree. The bug is real but locally consistent — which means downstream code cannot detect it from cards.json alone.

7. **Summary-enriched chunking is compromised where it's most needed.** The plan's §1.7 prepends card summary before embedding each chunk. Sections with missing summaries are disproportionately deep (L3–L4) where disambiguation matters most for retrieval; 292/684 (43%) of L3+L4 cards have no summary. Effectively a third of the knowledge base will embed without the disambiguating prefix the plan explicitly called for.

8. **`field_index.json` stores full field copies, not references.** Each acronym maps to a list of full field dicts duplicated from `fields.json`. No dangling pointers (good), but any edit to `fields.json` silently diverges from `field_index.json`. This is a drift hazard, not a current integrity failure. Plan (§1.4) is silent on which is the source of truth.

9. **Keywords pipeline is ambiguous.** Plan §1.6 shows `keywords: []` in the example with no population rules. Empty for the same 364 sections as `summary`. Populated cards carry 5–10 keywords (mode 8). Appears to be a free LLM side-output, not documented. Flag as ambiguous per the scope brief.

10. **Single-doc corpus.** All 1041 cards tag `NVM Express Base Specification @ 2.1`. Plan §1.1 says "base spec, transport specs"; transport specs haven't been ingested. Not a bug against what's there, but the acceptance bar "covers the spec" is not met at the corpus level.

## Accuracy estimate

- **cards.json schema conformance:** 100%. Shape is rigorous; no drift in keys or types.
- **cards.json content-correctness (title + parent):** **~90%**, bounded from above by `toc.json`'s 90.2% title-correctness ceiling since card titles are copy-forwarded from the TOC. Additional loss: the dup-`5.3.1` collapse removes ~0.1%; the ~4 exact phantom-title children are already counted in the TOC's error budget.
- **cards.json summary+keywords coverage:** **65%**. Flat gap; not an accuracy issue, a completeness issue.
- **cards.json cross-reference fidelity:** Tables refs 100% resolvable but only ~28% complete relative to the structural truth in `relationships.json`. Prose_blocks 100% in-bounds. Card-embedded relationships 100% consistent with the relationships file.
- **Cross-artifact integrity (set-diff / ID resolution):** **~96%** usable. 100% for fields↔tables and toc↔cards↔prose section sets. Degraded by the 3.9% relationship orphan rate and the 71.7% undercount in `cards.tables`.
- **Overall trustworthiness against §1.8 bar ("every table field extracted correctly, every cross-reference captured, no misattributed content"):** **Not met.** Misattribution is present (phantom cards, §3.2 shift, dup-collapse); cross-references have measurable orphan rate; some figures referenced in the text are genuinely missing from `tables.json`.

### TOC-bug propagation estimate

Taking the ~262 problematic TOC entries identified in `02_toc_accuracy.md` (102 body-text-promoted + 60 §3.2/§3.3 shift + 7 missing `1.4.1.x` keywords + 1 dup + ~8 phantom annex-B + ~80 other hierarchy artifacts):

- **Title-level propagation:** ~100%. Cards copy TOC titles verbatim. Every TOC title bug is a card title bug.
- **Structural propagation:** ~100%. parent/child chains in cards.json are internally consistent, which means the wrong chains from toc.json transfer unchanged.
- **Prose/tables propagation:** Prose is attached by section_id (and `prose.json` mirrors toc 1:1), so all prose gets attributed to whatever title the TOC lied about. Tables are further hurt by Issue #3 regardless.
- **Relationship propagation:** ~1–4%. Only relationships touching the buggy sections are affected; most of the 3.9% orphan rate is from *separate* deep-section and missing-figure gaps, not TOC bugs.
- **Net:** Roughly **20–25% of cards are downstream-corrupted by TOC bugs** (262 / 1041), mostly in §3.2, §3.3, §1.4.1, and annexes. Confidence: medium — depends on whether one counts "body-text-promoted but structurally consistent" entries as corruption or as a parallel anchor system.

## Recommended fixes (ordered by impact)

1. **Rebuild `toc.json` from the PDF bookmark outline with `PAGE_OFFSET = 24`** (per `02_toc_accuracy.md` recommendation) before any further cards/prose work. Fixes issues 5, 6, and the majority of propagation damage.
2. **Single-source `cards.tables` from `relationships.json`'s `contained_in` edges.** This fixes Issue #3 immediately and raises table-linkage from 28% to 100% of the structural truth.
3. **Re-run summary generation for the 364 empty cards and replace `cards_state.processed` with a per-section status record** (`attempted / succeeded / failed`, model, timestamp, error). Gates §1.7 chunking quality.
4. **Audit the 131 missing-figure references.** Extract list of unique missing figure numbers, run the table parser again over those page ranges specifically. Each one is a genuine content gap, not a naming gap.
5. **Resolve dup `5.3.1` at the TOC layer** by synthesizing a suffix (`5.3.1a` / `5.3.1b`) or promoting "Create I/O Completion Queue command" to `5.3.1` (its correct identity in the authoritative outline) and renumbering the current `5.3.1` to the correct anchor. Cards should follow automatically.
6. **Drop the 6 phantom annex-B cards** (`B.5.1.1`, `B.6.1.1`, `B.6.1.2`, `B.6.2.1`, `B.6.4.1`, `B.6.4.2`) or explicitly mark them `phantom: true` so retrieval can filter them.
7. **Decide field_index.json ownership:** either convert to pointer-based references into `fields.json`, or treat `field_index.json` as a read-only build artifact rebuilt from `fields.json` on every parse. Document which is source of truth.
8. **Document keyword provenance** in the plan or parser docs: is it LLM-generated from the same call as `summary`? If yes, the 1:1 empty-set correlation makes sense; if no, this is a real bug.
