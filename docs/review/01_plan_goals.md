# Plan Goals for specGPT Data Artifacts

Extracted from `BUILD_PLAN_FINAL.md`. Phase 1 is the source of truth for parser/data quality; Phase 2+ consumes this output.

## Plan goals for data

- **Foundational framing (Phase 1 opener):** "Parser quality directly determines answer quality. Get this right before moving forward."
- **Overall exit bar (§1.8):** "Parser is not done until output is trustworthy across the spec." Must verify: "every table field extracted correctly, every cross-reference captured, no misattributed content."
- **Week 1 gate (checklist):** "Tables parse cleanly for 10+ diverse spec sections."
- **Week 2 gate (checklist):** "Parser runs on entire spec with validated output."
- **Week 3 gate (checklist):** "Graph connects known concepts, search returns relevant results."
- **Hallucination stance (risk table):** "Never hallucinate" — if content is not in the indexed spec, the system must say so. Implies the data layer must expose gaps honestly rather than silently drop content.
- **Reliability hierarchy (risk table):** Structural extraction is the "reliable base"; LLM extraction is "supplementary" and must be flagged with lower confidence.

## Per-artifact acceptance bars

- **toc.json (§1.1):** Full spec segmented "by heading hierarchy using the table of contents." Must preserve section numbering, nesting depth, and document metadata (spec version, revision date, title, document type).
- **tables.json (§1.2 — "Most Critical Step"):** Each row captures field name, byte/bit offset, size, valid values, mandatory/optional status, description. Each table captures figure number, caption, parent section ID, column headers. Raw text representation stored alongside structured JSON. Edge cases explicitly called out as must-fix: "garbled columns, merged cells, footnotes in tables, multi-page tables that split across pages." Validation: "spot-check parsed tables against the actual spec PDF for diverse section types — log pages, command CDW layouts, identify structures, feature definitions, register maps."
- **fields.json / field_index.json:** Not named directly in the plan; implied by §1.2 row schema and §1.4 containment ("Fields belong to structures, structures belong to sections — captured automatically during table parsing").
- **prose.json (§1.3):** All non-table prose extracted per section; paragraph boundaries preserved; `shall` / `should` / `may` tagged with full text + section reference.
- **definitions.json (§1.3):** Acronym/definitions table parsed into `{term: definition}` lookup.
- **relationships.json (§1.4 — deterministic):** Regex-detected cross-references ("see Section X.Y.Z", "as defined in Figure N", "refer to Table M") emitted as `{source, target, type}`. Containment and figure↔section associations captured.
- **relationships_llm.json (§1.5):** Implicit relationships Haiku extracts beyond regex; global reconciliation merges duplicate entities and standardizes names; merged with structural set. Must be flagged as lower-confidence (per risk table).
- **cards.json (§1.6):** One card per section matching the fixed schema (section_id, title, spec_document, spec_version, summary, relationships, keywords, parent_section, child_sections, tables, prose_blocks). Summary generated by a free LLM.
- **Chunks (§1.7, not in data/ list):** ~500-token segments with 50-token overlap, metadata attached (section number, title, spec version, normative strength, parent card ID), card summary prepended before embedding.
- **Graph validation (§2.1):** "For known concepts, verify the graph connects all the pieces an engineer would need."

## Gaps / vague criteria

- **No numeric thresholds anywhere.** "Trustworthy," "cleanly," "validated," "relevant" — no precision/recall targets, no allowable error rate per artifact. "10+ diverse sections" is the only quantitative bar, and it's a sampling floor, not an accuracy floor.
- **"Diverse sections" undefined.** §1.2 lists five table types to cover, but no count-per-type or coverage percentage.
- **Field-level schema ambiguity.** §1.2 lists required row attributes but doesn't specify what to do when the PDF omits one (e.g., missing "valid values" column) — no null vs. drop vs. infer policy.
- **Cross-reference completeness unmeasured.** §1.4 says "every cross-reference captured" but provides no corpus-level sampling method.
- **Normative tagging coverage undefined.** §1.3 requires tagging `shall/should/may` but doesn't specify handling of negations, nested clauses, or non-normative uses of "shall."
- **Definitions scope unclear.** §1.3 says "the acronym/definitions table" — singular — but the spec has multiple glossaries across documents; merge policy unspecified.
- **LLM-relationship noise threshold.** Risk table acknowledges noise but sets no rejection rate or confidence floor.
- **Cards schema drift.** §1.6 example shows fields (`keywords`, `prose_blocks`) with no definition of how they're populated.
- **No end-to-end accuracy metric tied to data.** Eval set (§2.3) measures retrieval+generation, not parser fidelity in isolation.
